# 동기화 검증 체크리스트

이 체크리스트는 직접 손으로 테스트해야 확인 가능한 시나리오들을 정리한 것입니다.
새로운 기능 추가 전, 또는 동기화 코드를 수정한 뒤에 한 번씩 돌려보세요.

각 시나리오는 **준비 → 동작 → 기대 결과**의 3단계로 구성되어 있어요.
실패한 항목은 본인 환경/계정/브라우저를 메모해두면 디버깅이 쉽습니다.

---

## 1. 단일 기기 — 기본 백업/가져오기

### 1-1. 빈 클라우드에 업로드
- 준비: 로그인 직후, 클라우드(Firestore)에 데이터 없음. 로컬에 실/도안/바늘/부자재/프로젝트 약간 있음.
- 동작: 설정 → [백업] 클릭 → 확인 → 진행.
- 기대:
  - 단계별 토스트가 "실 → 도안 → 바늘 → 부자재 → 프로젝트" 순으로 한 번씩 뜬다.
  - 마지막 결과 카드에서 ↑(업로드) 카운트가 로컬 항목 수와 일치, ↓(다운로드)는 0.
  - 실패(×)가 없어야 함.

### 1-2. 빈 로컬에서 가져오기 (시크릿 모드 추천)
- 준비: 시크릿 창에서 같은 계정으로 로그인. 로컬 IndexedDB는 비어 있음.
- 동작: 설정 → [가져오기] 클릭 → 확인 → 진행.
- 기대:
  - 단계별 토스트가 entity 별로 한 번씩.
  - 결과 카드의 +(추가) 카운트가 클라우드 항목 수와 일치, ↻(업데이트)는 0.
  - 프로젝트 페이지에 가서 연결된 실/도안/바늘/부자재가 모두 잘 나오는지 눈으로 확인.
  - 단수 카운터 / 게이지도 함께 잘 가져왔는지 확인.

### 1-3. 변경 없음
- 준비: 1-1 또는 1-2 직후, 데이터 변경 없음.
- 동작: [백업] 또는 [가져오기] 다시 클릭.
- 기대: 모든 항목이 "변경없음(·)"으로만 잡힘. ↑/↓ 또는 +/↻은 0.

---

## 2. 양방향 — 두 기기 시나리오

### 2-1. 노트북에서 추가, 핸드폰에서 받기
- 준비: 노트북·핸드폰 모두 로그인.
- 동작:
  1. 노트북에서 새 실 1개, 새 도안 1개 추가.
  2. 노트북에서 [백업].
  3. 핸드폰에서 [가져오기].
- 기대: 핸드폰에 새 실/도안이 그대로 들어와 있음. cloudId 누락이 없으니 중복 안 생김.

### 2-2. 양 기기에서 동시에 다른 항목 추가
- 준비: 노트북 A 항목, 핸드폰 B 항목 추가 (동기화 전).
- 동작: 노트북 [백업] → 핸드폰 [백업] → 핸드폰 [가져오기] → 노트북 [가져오기].
- 기대: 양쪽 모두 A, B 둘 다 보유. 중복 없음.

### 2-3. 같은 항목을 양쪽에서 동시에 수정
- 준비: 같은 yarn 1개를 노트북에서는 색상 변경, 핸드폰에서는 메모 변경 (동기화 전).
- 동작: 노트북 [백업] → 핸드폰 [백업].
- 기대: updatedAt 이 더 최신인 쪽이 이김 (현 정책: last-write-wins). 어떤 쪽이 이겼는지 명확히 보이는지.
- 주의: 이 정책은 의도된 단순화. 충돌 병합이 필요하면 향후 별도 설계 필요.

---

## 3. 프로젝트 내부 동기화

### 3-1. 단수 카운터만 수정한 경우
- 준비: 프로젝트 1개에 카운터 1개. 백업 완료 상태.
- 동작: 카운터 +1을 5회 누르고 [백업].
- 기대: 프로젝트가 "↑ 1" 로 잡혀야 함 (카운터만 바뀌어도 프로젝트가 동기화 대상에 들어가야 함).

### 3-2. 게이지만 수정한 경우
- 준비: 프로젝트에 게이지 1개. 백업 완료 상태.
- 동작: 게이지 메모만 수정 후 [백업].
- 기대: 프로젝트가 "↑ 1" 로 잡혀야 함.

### 3-3. 연결된 실/도안 변경
- 준비: 프로젝트에 yarn link 1개. 백업 완료.
- 동작: yarn link 의 사용량(usedGrams) 변경 → [백업] → 다른 기기 [가져오기].
- 기대: 다른 기기에서 해당 프로젝트의 yarn link 가 새 값으로 반영. cloudId 가 보존되어 중복 link 가 생기지 않음.

## 4. 자동 백업

### 4-1. 자동 백업 끔
- 준비: 자동 백업을 "끔"으로 설정. 데이터에 변경 사항 있음.
- 동작: 페이지 새로고침.
- 기대: 4초 후 어떤 자동 토스트도 뜨지 않음. 수동 [백업]으로만 동작.

### 4-2. Wi-Fi에서만 — 데이터 환경
- 준비: 핸드폰을 모바일 데이터로만 연결 (Wi-Fi 끔). 자동 백업 모드 "Wi-Fi에서만".
- 동작: 페이지 새로고침.
- 기대: 자동 백업이 트리거되지 않음 (조용히 스킵).

### 4-3. 항상 자동 백업
- 준비: "항상 자동 백업" 모드. 변경 사항 있음.
- 동작: 페이지 새로고침.
- 기대: 약 4초 후 "자동 백업 완료 · X건이 클라우드와 동기화됐어요" 토스트가 뜸.
- 변형: 변경 사항이 0이면 토스트 없이 조용히.

### 4-4. 자동/수동 동시 트리거 방지
- 준비: "항상 자동 백업" 모드 + 변경 있음.
- 동작: 페이지 새로고침 후 4초 안에 [백업] 버튼 클릭.
- 기대:
  - 둘 중 먼저 시작한 쪽이 진행됨.
  - 두 번째는 "다른 동기화가 진행 중이에요. 끝난 뒤 다시 눌러주세요." 토스트.
  - 데이터 중복이나 손실 없음.

---

## 5. 엣지 케이스

### 5-1. 시크릿 모드 로그인 후 가져오기
- 준비: 시크릿 창. 같은 구글 계정으로 로그인.
- 동작: [가져오기].
- 기대: 클라우드 데이터가 그대로 내려옴. 이미지/사진은 안 옴 (1차 정책).

### 5-2. 다른 브라우저(Firefox, Safari) 로그인
- 준비: 평소 안 쓰던 브라우저로 로그인.
- 동작: [가져오기] → 데이터 확인 → 카운터 +1 → [백업].
- 기대: 정상 동작. 단, "Wi-Fi에서만" 자동 백업은 Firefox/Safari에서는 navigator.connection 없어서 트리거 안 될 수 있음 (의도된 보수 정책).

### 5-3. 오프라인 상태 백업 시도
- 준비: 비행기 모드 또는 네트워크 끔.
- 동작: [백업] 클릭.
- 기대: Firestore 호출이 실패하거나 큐에 쌓임. 토스트로 오류 안내. 로컬 데이터는 손상 없음.

### 5-4. JSON 백업 / 복원
- 준비: 로컬 데이터 있음.
- 동작: 로컬 파일 백업 → JSON 내보내기 → 다른 환경에서 JSON 가져오기.
- 기대: 모든 entity 복원. cloudId 도 함께 복원되어 이후 클라우드 백업 시 중복 안 생김.

---

## 6. 데이터 무결성 점검 (개발자 콘솔)

브라우저 개발자 도구 콘솔에서 다음을 한 번씩 실행해 cloudId 누락/중복이 없는지 확인할 수 있어요.

```js
// cloudId 누락된 항목 찾기
['yarns','patterns','needles','notions','projects','rowCounters','projectGauges'].forEach(async t => {
  const all = await window.dexie?.[t]?.toArray() || [];
  const missing = all.filter(x => !x.cloudId);
  console.log(`${t}: 총 ${all.length}, cloudId 누락 ${missing.length}`, missing);
});
```

(window.dexie 가 노출돼 있지 않으면 `import { db } from '/src/lib/db'` 후 사용)

문제 발견 시: 해당 entity 의 cloudId, updatedAt 값을 콘솔에서 확인해서 issue 로 남기면 디버깅이 쉽습니다.
